This is BedRock's **NORMALIZE â†’ OSM-ready** stage. It takes the four acquired sources

remains the project's downstream roadmap item.

## Pipeline

```bash
python3 normalize.py     # 1. reproject every in-scope layer -> normalized/<source>/<layer>.geojson (EPSG:4326)
python3 to_osm.py        # 2. tag + convert -> osm/<source>-<group>.osm.pbf  (one file per logical layer)
python3 assemble.py      # 3. osmium sort + merge -> osm/combined.osm.pbf
python3 verify_osm.py    # 4. per-layer counts + name:en/name:am coverage -> report.csv
# optional union with a Geofabrik extract (external input, not committed):
python3 assemble.py ~/ethiopia-latest.osm.pbf   # -> osm/with_osm.osm.pbf
```

Each script accepts an optional source filter, e.g. `python3 to_osm.py edas`.
`normalize.py` is idempotent (skips existing outputs; `--force` to rebuild).

## Output: one file per logical layer

Features are routed to a themed output file `osm/<source>-<group>.osm.pbf`. A source's tagging
module may define the grouping; otherwise features are routed by their primary OSM key
(`roads` / `buildings` / `landuse` / `places` / `boundaries` / `pois`). So small sources stay in
one file and large sources split into several:

| Source | Files |
|--------|-------|
| `cbebank` | `cbebank-pois` (ATMs) |
| `fayda` | per venue: `fayda-tele`, `fayda-bank`, `fayda-post_office`, `fayda-dars`, `fayda-mor`, `fayda-crrsa`, `fayda-palace_parking` |
| `edas` | `edas-roads`, `edas-buildings`, `edas-landuse`, `edas-places`, `edas-pois` |
| `ethionsdi` | `ethionsdi-roads`, `ethionsdi-places`, `ethionsdi-boundaries`, `ethionsdi-pois` |

## Multilingual names

Every named feature gets `name:en` and `name:am` (plus `name = name:am or name:en`). Method:
1. **Extract** native names where present â€” Fayda packs both languages in one field
   (`split_bilingual()` classifies each part by Unicode block: Ethiopic `U+1200â€“137F` â†’ `am`).
2. **Machine-translate** the bounded vocabulary (feature types, operators, admin units, common
   nouns) via the reviewable `lookups/am_vocab.json`.
3. **Transliterate** residual proper-noun tokens Latin â†’ fidel (`translit_lat2eth`) for full
   coverage. **Transliterated `name:am` is approximate** â€” `verify_osm.py` reports how much of
   each layer's `name:am` came from data vs vocabulary vs transliteration.

> Note: the EDAS `landmark` Amharic field (`legal_name`) is literally `"???"` in the source â€”
> the Amharic was destroyed before publication â€” so it is **not** used; landmark `name:am` is
> machine-generated like the rest.

## OSM tagging (representative)

Per-source rules live in `tagging/` + `lookups/`:

- **CBE ATM** â†’ `amenity=atm` + `operator`/`brand` (`Commercial Bank of Ethiopia`, `:am`) + `ref`.
- **Fayda** â†’ `office=government` + `government=register_office` + `operator` + `fayda:venue`.
- **EDAS landmark** â†’ by `fun_type` (`amenity`/`tourism`/`shop`/place-of-worshipâ€¦).
- **EDAS roads** â†’ `highway=*` (provisional, see caveat) + `oneway`/`lanes`/`maxspeed`.
- **EDAS buildings** â†’ `building=yes`; **LULC** â†’ `landuse`/`natural` (multipolygons).
- **EDAS cities / EthioSDI towns & gazetteer** â†’ `place=*`; **nh_aoi** â†’ `place=neighbourhood`.
- **EthioSDI** `58_road_2` uses its existing `highway` field; schools â†’ `amenity=school`;
  woreda/zone/region â†’ `boundary=administrative` + `admin_level`; cultural features, place
  markers, health â†’ `amenity=*`.

## Data caveats

- **CBE ATM positions are approximate (~1 km)** â€” every ATM carries `fixme=...` and
  `source:position=approximate`. The source rounds coordinates and stacks many ATMs on one point.
- **Fayda low-precision points are dropped.** `tagging/fayda.accept()` rejects any point whose
  lon or lat has fewer than 5 decimal places (rounded source coordinates) â€” 171 of 2,251 removed,
  and the `palace_parking` venue empties out entirely so no `fayda-palace_parking.osm.pbf` is
  produced. The source `sources/fayda/` is left untouched (full provenance).
- **EDAS road classification is provisional** â€” EDAS ships no data dictionary, so the numeric
  `functional` code â†’ `highway` mapping in `edas_road_class.json` is a best-effort guess; the raw
  codes are preserved on every way as `edas:functional` / `edas:type` / `edas:structural`.
- **Excluded from OSM conversion**: EDAS cadastral parcels (~1.15M, not OSM-appropriate), the
  `driving_direction` routing aux layer, and the identical `cities_Am`/`cities_Or`/`landmark_Am`
  Latin-duplicate variants.
- Some source shapefiles carry a garbage `Â±1.79e308` coordinate and a CP1252 `.dbf` with a bad
  `.cpg`; `normalize.py` handles both (world-bbox clip pre-reproject + `-oo ENCODING=CP1252`).
- A few source features still have **swapped/out-of-region coordinates** (the combined bbox
  reaches lat â‰ˆ 39, impossible for Ethiopia) â€” these survive the world-bbox clip and are left
  as-is for the downstream conflation/cleanup step to detect and fix.
